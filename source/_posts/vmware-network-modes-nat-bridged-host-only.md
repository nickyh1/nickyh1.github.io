VMware 虚拟机常见的三种网络模式是 Bridged、NAT 和 Host-only。它们看起来只是网卡模式不同，本质上却决定了虚拟机在网络里的身份：是像真实机器一样接入局域网，还是通过宿主机共享上网，还是只和宿主机组成一个隔离小网络。

<!-- more -->

## 核心结论

三种模式可以这样理解：

| 模式 | 虚拟机在网络里的身份 | 能否访问外网 | 外部能否主动访问虚拟机 | 典型用途 |
|---|---|---|---|---|
| Bridged 桥接 | 像真实物理机一样接入物理网络 | 可以 | 可以，只要网络互通 | 让虚拟机和局域网其他机器互访 |
| NAT | 在私有网段里，通过宿主机地址出去 | 可以 | 默认不可以，需要端口映射 | 虚拟机上网、测试环境 |
| Host-only | 只和宿主机组成私有网络 | 默认不可以 | 外部网络不可以 | 宿主机和虚拟机通信、隔离实验 |

简单判断：

```text
想让虚拟机上网：NAT
想让局域网访问虚拟机：Bridged 桥接
只想宿主机和虚拟机互通：Host-only
```

## Bridged：虚拟机像一台真实机器

桥接模式可以理解为：

```text
虚拟机网卡 <——> 物理交换机 / 路由器
```

虚拟机直接接入真实局域网，像一台独立物理机一样获取 IP。

例如：

```text
宿主机：192.168.1.100
虚拟机：192.168.1.101
网关：192.168.1.1
```

这时虚拟机和宿主机、局域网内其他服务器都在同一个真实网段里。只要防火墙和路由允许，其他机器也可以主动访问虚拟机。

桥接模式最接近真实服务器环境，适合做需要被局域网访问的测试机、服务端实验环境，或者模拟真实部署。

## NAT：虚拟机通过宿主机出去

NAT 模式可以理解为：

```text
虚拟机私有网络 ——> 宿主机 NAT ——> 外部网络
```

虚拟机在 VMware 创建的私有网段里，例如：

```text
虚拟机：192.168.100.10
宿主机：192.168.1.100
```

当虚拟机访问外网时，VMware 会做地址转换：

```text
源地址：192.168.100.10
转换成：192.168.1.100
```

外部服务器看到的是宿主机在访问它。响应回来时，也只需要回给宿主机 `192.168.1.100`。宿主机根据 NAT 连接表，再把响应转回虚拟机。

所以 NAT 模式下虚拟机可以访问外网，但外部机器默认不能主动访问虚拟机。如果需要外部访问虚拟机里的服务，一般要配置端口映射。

## Host-only：只和宿主机组成私有网络

Host-only 模式可以理解为：

```text
虚拟机 <——> 宿主机虚拟网卡
```

它会创建一个只存在于宿主机和虚拟机之间的私有小网络，例如：

```text
宿主机 VMnet1：192.168.56.1
虚拟机：192.168.56.10
```

这时虚拟机默认只能访问：

```text
宿主机
同一个 Host-only 网络里的其他虚拟机
```

它不能直接访问真实局域网，也不能直接访问互联网。外部网络也不知道这个私有网段的存在。

Host-only 的价值是隔离。比如你想搭一个不会影响真实局域网的实验环境，或者只需要宿主机和虚拟机通信，就很适合用 Host-only。

## 为什么 Host-only 会出现“出去但回不来”

这个现象的核心原因是：Host-only 网络是私有隔离网段，外部网络没有到这个网段的回程路由。

假设：

```text
虚拟机：192.168.56.10
宿主机物理网卡：192.168.1.100
外部服务器：192.168.1.200
```

如果虚拟机尝试访问外部服务器：

```text
192.168.56.10 ——> 192.168.1.200
```

流量可能通过宿主机的某些配置“发出去了”。但外部服务器回包时会看到：

```text
我要回给 192.168.56.10
```

问题是，真实网络并不知道：

```text
192.168.56.0/24 这个网段在哪里？
```

于是响应包找不到回程路径，结果就是：

```text
请求出去了，但响应回不来
```

这不是 Host-only 天生支持单向通信，而是缺少回程路由，或者没有做 NAT 转换。

## NAT 为什么能回来

NAT 和 Host-only 的关键区别在于，NAT 会把虚拟机的私有地址转换成宿主机地址。

NAT 模式下：

```text
虚拟机 192.168.100.10 访问外部
转换成宿主机 192.168.1.100 访问外部
```

外部服务器回包时只需要回给：

```text
192.168.1.100
```

宿主机收到响应后，再根据 NAT 表把流量转发回虚拟机。

Host-only 默认不做这个转换。如果虚拟机以 `192.168.56.10` 作为源地址直接出去，外部网络不知道怎么回到这个地址，所以响应就回不来。

## 如果必须让 Host-only 访问外部怎么办

一般不建议把 Host-only 改造成上网模式，因为它的定位就是隔离。但如果确实要这么做，需要额外配置：

```text
宿主机开启路由转发
宿主机配置 NAT / 共享网络
外部网络添加到 Host-only 网段的静态路由
防火墙放行对应流量
```

本质上有两条路：

第一，让宿主机给 Host-only 网段做 NAT。这样外部只看到宿主机地址，回包能回到宿主机，再由宿主机转回虚拟机。

第二，让外部网络知道 `192.168.56.0/24` 在哪里，也就是加静态路由。但这通常要改路由器或上游网络配置，实验环境里不太方便。

## 我的理解

我之前容易把这三种模式理解成“能不能上网”的区别。现在更准确的理解是：它们的区别在于虚拟机被放进了哪个网络里。

```text
Bridged：虚拟机进入真实局域网
NAT：虚拟机进入私有网段，但通过宿主机代理出去
Host-only：虚拟机进入只和宿主机互通的隔离网段
```

Host-only 下所谓“出去回不来”，不是因为网络只能单向走，而是因为外部网络不知道如何返回 Host-only 私有网段。网络通信必须有去程和回程，少了回程路由，就会表现成请求发出去了但没有响应。

## 一句话总结

Bridged 让虚拟机像真实机器一样接入局域网，NAT 让虚拟机通过宿主机共享访问外部网络，Host-only 只让虚拟机和宿主机组成隔离私有网络；Host-only 下流量“出得去、回不来”，本质是外部网络没有到 Host-only 私有网段的回程路由，也没有 NAT 帮它做地址转换。
